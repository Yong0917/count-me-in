# PRD — 참석여부 공유앱 (count-me-in)

> 버전: v0.1 (MVP) · 작성일: 2026-06-11 · 상태: 초안

---

## 1. 개요 / 배경

### 배경 (문제 정의)
초보자 5명이 함께 운동을 치는 소모임에서, 모임 때마다 **"이번엔 누가 와요?"** 를
단톡방에서 일일이 물어보고, 답이 흩어지고, 결국 누가 오는지 헷갈리는 일이 반복된다.
참석 여부를 한 곳에서 **달력으로 한눈에** 보고 싶다.

### 한 줄 제품 정의
> **참가 코드 하나로 같은 페이지에 모여, 달력에서 오늘 누가 오는지 바로 확인하고 내 참석을 탭 한 번으로 표시하는 소모임용 참석 공유 웹앱.**

### 핵심 가치
- **간편함**: 회원가입 없이 **참가 코드 입력(또는 링크 탭)** → 이름 선택 → 바로 사용
- **한눈에**: 페이지를 열면 **디폴트로 오늘 참석 현황**이 보이고, 달력의 다른 날짜를 누르면 그날 현황으로 전환
- **모여있음**: 같은 참가 코드를 가진 사람들끼리 하나의 페이지를 공유
- **가벼움**: 5명 내외 소모임에 딱 맞는 최소 기능

## 2. 목표와 성공 지표

### 정성 목표
- 모임 전 "누가 오는지" 묻는 단톡방 대화를 없앤다.
- 멤버가 30초 안에 자기 참석 여부를 표시할 수 있다.

### 성공 지표 (측정 가능)
| 지표 | 목표 |
| --- | --- |
| 모임당 참석 응답률 (멤버 중 응답한 비율) | ≥ 80% |
| 링크 접속 → 참석 표시까지 소요 시간 | ≤ 30초 |
| 주 1회 이상 앱을 여는 멤버 비율 | ≥ 70% |
| 모임 전날 기준 "미정" 비율 | ≤ 20% |

## 3. 타깃 사용자 / 페르소나

| 역할 | 설명 | 핵심 니즈 |
| --- | --- | --- |
| **주최자(운영자)** | 모임을 만들고 일정을 등록하는 1명 | 일정 등록, 참석 현황 파악 |
| **멤버** | 함께 운동하는 초보자 4~5명 | 빠른 참석 표시, 다른 사람 참석 확인 |

- **공통 특성**: 앱 사용에 익숙하지 않을 수 있는 초보자. 복잡한 가입/설정에 거부감.
- **사용 환경**: 대부분 **모바일**에서 단톡방 링크를 눌러 접속.

## 4. 핵심 사용자 시나리오 (User Stories)

1. **(주최자)** 나는 그룹을 만들고 멤버 이름을 등록해, **참가 코드(와 링크)** 를 단톡방에 뿌리고 싶다.
2. **(주최자)** 나는 매주 화·목 저녁 같은 **정기 일정**과, 가끔 잡히는 **번개 모임**을 둘 다 등록하고 싶다.
3. **(멤버)** 나는 **참가 코드를 입력하거나 링크를 눌러** 들어가서, **내 이름을 한 번 선택**하면 다음부터는 바로 내 화면이 떴으면 좋겠다.
4. **(멤버)** 나는 페이지를 열자마자 **오늘 누가 오는지**가 바로 보였으면 좋겠다.
5. **(멤버)** 나는 달력에서 **다른 날짜를 누르면 그날 참석 현황**으로 바뀌었으면 좋겠다.
6. **(멤버)** 나는 달력에서 **참석 / 불참 / 미정**을 탭 한 번으로 바꾸고, "30분 늦어요" 같은 **한줄 메모**를 남기고 싶다.

## 5. 기능 요구사항 (MVP)

| ID | 기능 | 설명 | 우선순위 |
| --- | --- | --- | --- |
| **F1** | 그룹 생성 & 참가 코드/링크 | 그룹명을 정하고 **사람이 입력 가능한 참가 코드**(예: 6자리)와 그 코드를 담은 공유 링크 발급 | P0 |
| **F2** | 코드 입장 & 이름 선택 식별 | 참가 코드 입력 **또는** 링크 접속으로 같은 페이지 진입 → 멤버 목록에서 본인 선택, localStorage에 기억(다음 접속 자동). 이름 추가/변경 가능 | P0 |
| **F3** | 일정 관리 | **정기**(요일 반복 규칙) + **비정기**(단발) 일정 생성/수정/삭제 | P0 |
| **F4** | 월간 달력 뷰 | 날짜별 참석 인원 요약 마커 표시, 오늘 강조, 모바일 반응형 | P0 |
| **F5** | 참석 토글 | **참석 / 불참 / 미정** 3상태를 탭으로 전환 | P0 |
| **F6** | 참석 현황 패널 (디폴트=오늘) | 페이지를 열면 **기본으로 오늘 일정의 참석·불참·미정 명단**을 표시. 달력에서 다른 날짜를 누르면 그날 현황으로 전환 | P0 |
| **F7** | 한줄 메모/댓글 | 일정별 코멘트 작성·표시 | P1 |

### 기능 상세

**F1. 그룹 생성 & 참가 코드/링크**
- 그룹명 입력 → 그룹 생성 → **참가 코드**(사람이 입력 가능한 6자리, 예: `ABC123`)와 그 코드를 담은 공유 링크(`/{join_code}`)를 함께 발급.
- 내부적으로는 추측 불가능한 긴 `share_token`을 함께 두어 데이터 접근 통제에 사용(코드는 사람용 별칭).
- "코드 복사" / "링크 복사" 버튼 제공.

**F2. 코드 입장 & 이름 선택 식별**
- 진입 경로 2가지: ① 첫 화면에서 **참가 코드 입력** ② **링크 탭**(코드 자동 적용). 둘 다 같은 페이지로 도달.
- 진입 후 멤버 명단을 보여주고 본인 선택 → `member_id`를 localStorage 저장.
- 다음 접속부터는 이름 선택 화면을 건너뜀("나 아니에요"로 재선택 가능).
- 명단에 없으면 "이름 추가"로 새 멤버 생성.

**F3. 일정 관리**
- **비정기**: 날짜·시간·장소·제목을 지정해 단발 일정 생성.
- **정기**: 요일 + 시간(+장소·제목)으로 반복 규칙 등록. 여러 요일은 규칙을 여러 개 등록.
- 누구나 일정을 만들 수 있되, 수정/삭제 권한 범위는 [열린 질문] 참고(MVP는 전원 가능으로 단순화).

**F4. 월간 달력 뷰 (홈)**
- 경량 커스텀 월간 그리드. **오늘 강조**, 좌우로 월 이동.
- 일정이 있는 날에 **참석 인원 수 / 전체** 형태의 요약 마커 표시.
- 모바일 우선 반응형(한 손 조작 가능한 탭 영역).
- 달력과 현황 패널(F6)이 한 페이지에 함께 보임. 날짜를 누르면 선택 날짜가 바뀜.

**F5. 참석 토글**
- 상태: `참석(going)` / `불참(not_going)` / `미정(maybe)`. 기본값은 미정(미응답).
- 현황 패널에서 본인 행의 버튼으로 즉시 전환, 변경 즉시 저장.

**F6. 참석 현황 패널 (디폴트 = 오늘)**
- **페이지 진입 시 선택 날짜의 기본값은 "오늘"** → 오늘 일정의 참석/불참/미정 명단을 바로 표시.
- 달력에서 다른 날짜를 누르면 해당 날짜의 현황으로 전환.
- 참석/불참/미정으로 그룹핑된 멤버 명단 + 인원 요약("참석 3 · 불참 1 · 미정 1").
- 오늘(또는 선택 날짜)에 일정이 없으면 "이 날 일정 없음 / 일정 추가" 안내.

**F7. 한줄 메모/댓글**
- 일정별로 멤버가 짧은 코멘트 작성. 작성자·시간 표시. (P1)

## 6. 비기능 요구사항

- **모바일 우선 반응형**: 360px 폭부터 자연스럽게 동작. 탭 타깃 최소 44px.
- **성능**: 초기 화면 로딩 체감 2초 이내(소규모 데이터 전제).
- **데이터 정합성**: 같은 멤버가 같은 일정에 참석 상태는 1개만(고유 제약).
- **접근성 기본**: 색만으로 상태 구분하지 않기(아이콘/라벨 병행), 명도 대비 확보.
- **단순성**: 5명 규모에 과한 인프라(큐/캐시/배치) 없이 운영.

## 7. 데이터 모델 (초안)

```
groups
  id            uuid (pk)
  name          text
  join_code     text (unique, 사람이 입력 가능한 6자리 별칭, 예: ABC123)
  share_token   text (unique, 추측 불가 긴 랜덤 — 실제 접근 통제용)
  created_at    timestamptz

members
  id            uuid (pk)
  group_id      uuid (fk → groups)
  name          text
  color         text            -- 달력/명단 식별용 색상
  created_at    timestamptz

recurring_schedules            -- 정기 일정 "규칙"
  id            uuid (pk)
  group_id      uuid (fk → groups)
  title         text
  weekday       int             -- 0~6, 요일 1개당 1행 (화·목이면 2행)
  start_time    time
  end_time      time
  location      text
  active        boolean

events                          -- 실제 일정 인스턴스
  id            uuid (pk)
  group_id      uuid (fk → groups)
  title         text
  event_date    date
  start_time    time
  end_time      time
  location      text
  source        text            -- 'recurring' | 'adhoc'
  schedule_id   uuid (nullable, fk → recurring_schedules)
  created_at    timestamptz

attendances
  id            uuid (pk)
  event_id      uuid (fk → events)
  member_id     uuid (fk → members)
  status        text            -- 'going' | 'not_going' | 'maybe'
  updated_at    timestamptz
  UNIQUE(event_id, member_id)

comments
  id            uuid (pk)
  event_id      uuid (fk → events)
  member_id     uuid (fk → members)
  body          text
  created_at    timestamptz
```

### 정기 일정 처리 = Lazy materialize
- `recurring_schedules`에는 **규칙만** 저장한다.
- 달력은 규칙으로부터 **가상 일정(occurrence)** 을 계산해 표시한다(실제 행 없이도 보임).
- **누군가 그 날짜에 참석을 표시하거나 메모를 다는 순간에만** 해당 날짜의 실제 `events` 행을
  생성한다(`source='recurring'`, `schedule_id` 연결).
- 효과: 별도 크론/배치 불필요, 아무도 응답 안 한 빈 일정 행이 쌓이지 않음, 구현 단순.

## 8. 화면 / 정보구조 (IA)

1. **코드 입장 화면** — 참가 코드 입력(또는 링크로 자동 적용).
2. **이름 선택 게이트** — 첫 진입 시 본인 선택(또는 추가). 이후 자동 통과.
3. **달력 + 현황 홈(한 페이지)** — 위/옆엔 월간 달력(참석 요약 마커), 아래/한쪽엔 **선택 날짜(기본=오늘)의 현황 패널**(명단 + 본인 토글 + 한줄 메모). 날짜를 누르면 패널이 그 날짜로 전환.
4. **일정 추가/관리** — 정기/비정기 일정 생성·수정·삭제.
5. **그룹 설정** — 그룹명, 멤버 관리, 참가 코드/링크 복사.

```
[코드 입장] → [이름 선택] → [달력 + 현황 홈 (디폴트=오늘 현황)]
                                 │   ↑ 달력 다른 날짜 탭 → 그날 현황으로 전환
                                 ├─ [일정 추가/관리]
                                 └─ [그룹 설정 / 코드·링크 공유]
```

## 9. 기술 아키텍처

- **프론트엔드**: Next.js(App Router) + React + TypeScript, Tailwind CSS(반응형).
- **달력 UI**: 외부 의존 최소화한 **경량 커스텀 월간 그리드**(MVP 충분).
- **백엔드/DB**: Supabase(PostgreSQL). 클라이언트에서 Supabase JS로 접근.
- **데이터 스코프**: 사용자가 입력한 `join_code`(또는 링크) → 서버에서 그룹의 `share_token`으로 매핑해 해당 그룹 데이터만 조회/변경.
- **RLS 정책(개요)**: 정식 인증이 없으므로, 접근은 `share_token` 보유 여부로 통제.
  - 권장 구현: 토큰 검증을 거치는 RPC/Edge Function 또는 토큰 일치 시에만 행을 노출하는 정책으로 그룹 데이터를 격리. (anon 키로 전체 테이블이 노출되지 않도록 주의)
- **배포**: Vercel(프론트), Supabase(관리형 DB). 환경변수로 Supabase URL/anon key 주입.
- **타임존**: Asia/Seoul 고정(가정).

## 10. 보안 / 프라이버시 고려

- **코드/링크 = 접근 권한**: 참가 코드(또는 링크)를 아는 사람은 누구나 그룹을 보고 참석을 표시할 수 있다.
  지인 5명 소모임 전제라 **민감도가 낮음**. 단, 짧은 `join_code`는 추측·대입(brute-force) 위험이 있으므로
  실제 데이터 접근 통제는 긴 `share_token`으로 하고, 필요 시 코드 입력 시도 제한(rate limit)을 둔다.
- **이름 사칭 가능성**: 로그인이 없어 다른 사람 이름으로 참석을 누를 수 있다. 낮은 위험으로 수용하되,
  필요 시 **후속으로 멤버별 4자리 PIN** 옵션 추가(범위 밖 참고).
- **데이터 격리**: anon 키 사용 시 RLS로 그룹 간 데이터가 섞이지 않도록 반드시 격리.
- **개인정보 최소화**: 이름 외 개인정보를 수집하지 않는다.

## 11. 범위 밖 (Out of Scope) / 후속 로드맵

MVP에서 **제외**, 향후 단계 후보:
- 리마인더 알림(웹푸시 / 카카오 알림톡)
- 참석 정원(최소·최대 인원) 및 마감 처리
- 멤버별 PIN 등 본인 확인 강화 / 정식 소셜 로그인(카카오)
- 참석 통계·출석률 리포트
- 장소 지도 연동, 사진/공지 등 부가 콘텐츠
- 여러 모임(다중 그룹) 한 화면 관리
